# アーキテクチャの概要

xyMusicUpdater は、Navidrome のコアストリーミング機能を置き換えることなく、それを補完するように設計されたハイブリッドアーキテクチャで構築されています。

## 構造

*   **Navidrome**: 高速スキャン、Subsonic API の互換性、およびエンドユーザークライアント（スマートフォン、Web プレーヤー）への軽量なオーディオストリーミングを処理します。
*   **xyMusicUpdater (Python/Django)**: 「マネージャー」ノードとして機能します。Web スクレイピング (`yt-dlp`)、API 統合 (MusicBrainz)、オーディオ処理 (`ffmpeg`)、および詳細なデータベース管理などの、複雑でリソースを消費するタスクを処理します。Navidrome の最新のプラグインシステムは意図的に使用していません。そのフレームワークの制限により、特定の高度な機能の実装が制限される可能性があるためです。

## データフローとストレージ

システムは、マネージャーと Navidrome の間でデータを渡すために、複数の共有 Docker ボリュームを利用します。

1.  **`/music/temp`**: メインの作業ディレクトリです。`yt-dlp` は生のオーディオをここにダウンロードします。Navidrome は新しい追加のためにこのフォルダーをスキャンします。
2.  **`/music/permanent`**: アーカイブディレクトリです。特定のプレイリストや手動操作によって保護された曲は、パージの実行中にここに移動されます。
3.  **`/navidrome_data`**: `navidrome.db` が含まれています。xyMusicUpdater は直接 SQLite クエリを使用してこのデータベースとやり取りし、Navidrome の読み取り専用 API をバイパスして、コンピレーションアルバムの統合や即時のメタデータ同期などの機能を実現します。
4.  **`/app/data`**: xyMusicUpdater の永続的な状態フォルダーです。`db.sqlite3`（Django モデル）、カスタム UI 背景、およびオーディオエディタ用の分離された `/previews` が格納されています。

## コアロジックモジュール (`backend/core/logic/`)

バックエンドは、わかりやすくモジュール化されています。

*   **`ytdlp.py`**: `yt-dlp` CLI をラップします。ダウンロード、出力のサニタイズ、およびローカルと Navidrome データベースの両方に対する重複チェックを処理します。
*   **`tagger.py`**: MusicBrainz および iTunes API と統合して、メタデータとカバーアートを取得します。`mutagen` を使用して ID3 タグをファイルに直接書き込みます。
*   **`editor.py`**: 正確なオーディオのトリミングのために `ffmpeg` をラップします。変更を確定する前に、`/app/data/previews` に安全に保存される一時的なプレビューを生成します。
*   **`pipeline.py`**: Cron ジョブと手動ダウンロードをオーケストレーションし、ダウンロード、タグ付け、登録のステップをリンクします。
*   **`storage.py`**: ストレージのクォータを管理し、保持ポリシーとプレイリストの保護に基づいてパージロジックを実行します。
*   **`navidrome.py`**: Navidrome の内部データベースとやり取りするための直接的な SQLite クエリ（再スキャンのトリガー、コンピレーションの統合など）が含まれています。
*   **`utils.py`**: Server-Sent Events (SSE) ブロードキャストや安全な構成の取得など、共有ユーティリティを提供します。
